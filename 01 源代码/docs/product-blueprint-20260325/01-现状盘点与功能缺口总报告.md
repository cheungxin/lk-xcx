# 现状盘点与功能缺口总报告

## 1. 分析范围

本报告基于以下内容整理：

- 当前小程序代码目录 `01 源代码/miniprogram`
- 当前已有分包结构、路由、页面、组件、Mock 数据、存储工具
- 你本轮补充的业务口径：行测、申论、每日打卡、自主刷题、模拟刷题、错题、收藏、记录、报告、我的

本报告不沿用旧 `docs` 结论，以下判断以当前代码现状为准。

## 2. 当前项目阶段判断

当前项目更像是“刷题产品原型工程”，不是“可交付的完整业务版本”。

判断依据：

- 大量页面数据来自 `mock/*.js` 或页面内常量。
- 首页、我的、训练入口中存在大量“即将开放 / 开发中”的占位逻辑。
- 业务接口层 `interface/api.js` 还是通用示例，不是刷题领域模型。
- 记录、错题、收藏主要依赖本地缓存，没有统一的业务写回逻辑。
- 申论没有真正落地页面体系。

## 3. 现有模块完成度盘点

### 3.1 工程与结构层

已具备：

- 主包与分包结构已经存在。
- 首页、我的、训练分包、课程分包、社区分包已经拆开。
- 路由封装已经有统一入口：`miniprogram/router/index.js`
- 本地存储工具已经封装：`miniprogram/utils/storage.js`

存在问题：

- 路由常量声明的页面和 `app.json` 实际注册页面不完全一致。
- `pages/exam-tab`、`pages/quick-tab`、`pages/memorize`、`pages/user-info` 存在目录，但未在 `app.json` 注册。
- 当前 TabBar 只有“首页、我的”两个正式 Tab，和产品规划不一致。

结论：

- 工程基座可用。
- 页面注册与真实导航结构需要先统一，否则后续所有业务迭代都会反复返工。

### 3.2 首页与学科入口

已具备：

- 首页已经有考试类型、地区、学科标签、Banner、快捷入口、专项练习树、模拟考试区块。
- 行测 / 申论两个一级学科标签已存在。
- 考点树组件基础可复用。

未完成：

- 每日一练入口只有占位，没有每日列表、没有日期维度任务流。
- 最新时政入口只有占位。
- 历年试卷入口只有占位。
- 套题练习入口只有占位。
- 专项练习点击后没有进入真实业务流。
- 首页模考“开始考试”仍是 Toast，没有跳到统一答题流程。
- 申论首页并未切换成独立业务内容，仍沿用行测式结构。

结论：

- 首页已经有“信息架子”，但还没有真正成为业务调度中心。

### 3.3 行测答题链路

已具备：

- 普通练习页 `subpackages/training/practice`
- 模考页 `subpackages/training/exam`
- 答题卡页 `subpackages/training/answer-card`
- 解析页 `subpackages/training/analysis`
- 数资数算辅助答题页 `subpackages/training/math`

已实现的能力：

- 基础答题、选项选择、翻题、标记、答题卡跳题。
- 练习缓存、结果缓存、基础解析展示。
- 模考页有时间倒计时、交卷、结果面板雏形。

缺口：

- 练习页与模考页各自维护一套状态，没有抽成统一答题引擎。
- 没有区分入口来源，无法知道本次作答属于“每日打卡 / 自主刷题 / 模拟考试”中的哪一类。
- 提交后只写本地缓存，没有完整的记录沉淀。
- 报告页不区分题型、考点树、用时分布、错题归因。
- 没有真正的“从入口配置页到统一答题页再到报告页”的标准闭环。

结论：

- 已有答题原型，但业务模式没有抽象清楚，是当前最关键的结构性缺口。

### 3.4 我的、记录、错题、收藏、报告

已具备：

- 我的首页原型。
- 练习记录页、错题本页、收藏页、统计页、设置页。

当前状态：

- 练习记录是页面内 Mock 时间线。
- 统计页是 Mock 图表。
- 错题本主要读取本地 `wrongQuestions`。
- 收藏主要读取本地 `favorites`。
- 我的首页菜单大多仍是 Toast 占位。

缺口：

- 没有统一记录中心。
- 没有按类型筛选、按时间筛选、按知识点 / 试卷筛选的完整逻辑。
- 没有从记录进入报告、从报告进入解析的完整跳转设计。
- 错题和收藏缺少详情列表与重练入口。
- 我的首页缺少行测 / 申论维度切换。

结论：

- “我的”模块现在只完成了视觉层第一步，没有完成数据闭环。

### 3.5 登录与用户体系

已具备：

- 登录页、静默登录工具、手机号授权登录壳子。

缺口：

- 登录仍然是 Mock 返回。
- Token 校验、用户资料拉取、学习档案初始化都未真正接后端。
- 登录成功后没有拉取用户学科偏好、历史统计、订阅提醒等个性化数据。

结论：

- 当前登录只能算联调前占位版本。

### 3.6 数据层与接口层

已具备：

- `request.js` 请求封装。
- `api-config.js` 环境配置。
- `interface/api.js` 通用接口示例。

缺口：

- 没有题库、试卷、考点树、每日任务、练习记录、错题、收藏、报告等领域接口。
- 当前接口命名与业务不匹配，不能支撑真实开发分工。
- 没有统一数据字典，前后端难以对齐。

结论：

- 接口层当前不具备直接开发业务的条件，必须先重构为领域接口。

### 3.7 申论模块

当前状态：

- 首页只有“申论”标签。
- Mock 考点树里有申论的五类题型节点。
- 没有申论首页。
- 没有申论材料阅读页。
- 没有申论作答页。
- 没有申论报告页。
- 没有申论热点素材、规范表述、自选题、随机题入口。

结论：

- 申论目前仍处于“概念存在，但业务未启动”的状态。

## 4. 当前最核心的缺口

如果只抓关键问题，当前最缺的不是单个页面，而是以下 8 个中枢模块。

### 4.1 统一业务模式定义

必须先定义三种核心模式：

- 每日打卡
- 自主刷题
- 模拟刷题

否则同一个答题页提交之后，系统不知道应该写到哪一类记录里，也无法生成正确报告。

### 4.2 统一答题上下文

建议把所有答题入口统一成一个上下文对象，例如：

```js
{
  subject: 'xingce',
  bizType: 'daily',
  sceneType: 'checkin',
  sourceType: 'daily_task',
  sourceId: 'daily-2026-03-25',
  title: '3月25日每日一练',
  questionIds: ['q1', 'q2', 'q3', 'q4', 'q5'],
  duration: 10,
  config: {},
}
```

这个上下文必须从入口页一路传到答题页、交卷页、报告页、记录页。

### 4.3 统一记录中心

所有提交行为都应该先沉淀为一条练习记录，再派生出：

- 统计数据
- 错题数据
- 收藏关系
- 报告展示
- 日历打卡

当前这层还没有建立。

### 4.4 模拟考试数据模型不够丰富

你提到的一点非常关键：模考接口的数据一定要丰富。

原因：

- 模拟考试不是简单题目列表，它还需要试卷、模块、题型、时间配置、用户作答详情、报告维度。
- 后面的练习记录、报告、错题归因、知识点统计都依赖这套数据。

### 4.5 行测与申论未分层设计

当前首页只是标签切换，但业务上两者完全不同：

- 行测偏客观题、快速反馈、统一判分。
- 申论偏材料题、文本作答、分题展示、二次批改 / AI 评价可能性更高。

如果不先分层，后续答题页会越做越乱。

### 4.6 我的模块缺少学科维度

后续“我的”必须支持至少两层维度：

- 学科维度：行测 / 申论
- 业务维度：打卡 / 自主刷题 / 模拟考试

否则统计和记录会混在一起，体验会很差。

### 4.7 页面流转标准没有固定

当前缺少统一路径定义，例如：

- 首页 -> 每日一练列表 -> 答题页 -> 报告 -> 解析 -> 返回列表
- 首页 -> 考点练习配置 -> 答题页 -> 报告 -> 记录
- 首页 -> 模考配置 -> 答题页 -> 报告 -> 解析 / 记录

这部分必须在开发前先固化。

### 4.8 组件资产虽然存在，但没有真正产品化

例如：

- `components/page/question-item` 还是空壳。
- `components/page/score-card` 还是空壳。
- 树组件可用，但没有沉淀为统一业务组件。

说明现在的组件层也还没有完成产品级封装。

## 5. 建议先定下来的产品结构

### 5.1 一级导航建议

建议先固定为两个正式底部 Tab：

- 首页
- 我的

原因：

- 当前项目已经按这个结构落地。
- 你的核心业务流是从首页进入各种训练模式，再由我的沉淀数据。
- 不建议现在再增加过多底部 Tab，否则会分散入口并增加维护成本。

### 5.2 首页二级结构建议

首页顶部先选定：

- 考试类型
- 地区
- 学科

然后根据学科切换不同内容：

- 行测首页：每日一练、时政、历年试卷、套题练习、模拟考试、考点树
- 申论首页：热点素材、规范表述、随机出题、自选题目、历史作答

### 5.3 答题页建议

不要按“页面名”拆逻辑，而是按“答题引擎 + 题型渲染器”来拆：

- 通用答题外壳：计时、答题卡、题号、收藏、标记、交卷、缓存
- 行测题渲染器：选择题、材料题、数资数算
- 申论题渲染器：材料阅读、文本作答、字数统计、草稿、提交

## 6. 推荐的开发前置决策

在任何大规模开发前，建议先定死以下 6 个决策：

1. 首页是否继续只保留“首页 / 我的”双 Tab。
2. 行测是否统一用一个答题页承接每日 / 自主 / 模拟三种入口。
3. 申论是否使用“统一外壳 + 独立渲染器”的方式，而不是完全复刻行测答题页。
4. 练习记录是否统一存一张主表，再通过 `bizType` 区分。
5. 我的模块是否默认支持学科切换。
6. 后端接口是否先按“行测优先、申论后补”节奏建设。

## 7. 本项目接下来最合理的推进方式

建议不要先补零散页面，而要按以下顺序推进：

1. 先补业务抽象与接口契约。
2. 再补行测三种模式闭环。
3. 再补我的数据沉淀体系。
4. 最后补申论体系。

原因很简单：

- 申论依赖更复杂的数据结构。
- 我的模块依赖前面所有练习写回逻辑。
- 如果前面没有统一上下文，后面所有页面都会返工。
