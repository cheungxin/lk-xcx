# 数据模型与接口契约建议

## 1. 文档目标

这份文档用于解决当前项目最大的问题之一：页面有原型，但没有稳定的业务数据契约。

后续无论是自己开发，还是把任务拆给其他 AI，都建议先遵守这份数据模型和接口分组。

## 2. 总体原则

### 2.1 页面只拿业务数据，不自己拼业务语义

例如答题页不应该自己判断这是不是每日打卡，而应该由接口或启动参数明确告诉它。

### 2.2 所有练习统一先落一条主记录

不管是行测还是申论，不管是每日、自主还是模考，都先沉淀一条练习主记录，再由系统派生：

- 报告
- 错题
- 收藏关联
- 统计
- 日历

### 2.3 本地缓存只做“临时状态”

当前 `storage.js` 里保存的 `practiceCache`、`practiceResult` 可以保留，但只应该承担：

- 暂存
- 断点续答
- 页面返回恢复

真正的业务数据不能长期依赖本地缓存。

## 4. 接口分组建议

### 4.1 首页与学科配置

- `GET /subject/home-config`
- `GET /subject/banner-list`
- `GET /subject/quick-entries`

### 4.2 行测题库与考点

- `GET /xingce/knowledge-tree`
- `GET /xingce/question/list`
- `GET /xingce/paper/list`
- `GET /xingce/paper/detail`

### 4.3 每日一练

- `GET /daily-task/list`
- `GET /daily-task/detail`
- `GET /daily-task/today`

### 4.4 自主刷题 / 模拟刷题启动

- `POST /practice/start`
- `POST /mock/start`

这两个接口建议返回统一启动结果：

```js
{
  sessionId: 'sess_xxx',
  answerContext: {},
  questionList: []
}
```

### 4.5 作答过程

- `POST /session/save-progress`
- `POST /session/submit`
- `GET /session/detail`
- `GET /report/detail`

### 4.6 我的相关

- `GET /record/list`
- `GET /wrong/list`
- `GET /favorite/list`
- `GET /stats/overview`
- `GET /stats/calendar`

### 4.7 申论相关

- `GET /shenlun/home-config`
- `GET /shenlun/material/list`
- `GET /shenlun/expression/list`
- `POST /shenlun/random/start`
- `POST /shenlun/custom/start`
- `POST /shenlun/session/submit`

## 6. 记录页筛选能力需要的接口参数

为了满足你提到的“按类型、按时间、按知识点或试卷筛选”的诉求，记录接口至少要支持：

```js
GET /record/list?subject=xingce&bizType=mock&startDate=2026-03-01&endDate=2026-03-25&keyword=判断推理
```

建议参数：

- `subject`
- `bizType`
- `startDate`
- `endDate`
- `keyword`
- `pageNo`
- `pageSize`

## 7. 为什么要先补接口契约

如果不先定义清楚数据模型和接口契约，接下来会发生三件事：

1. 页面先写了，但后面换接口会整体返工。
2. 不同 AI 写出来的数据字段互相不兼容。
3. 记录、报告、错题无法共享一条主链路。

所以建议任何页面开发前，优先以这份文档作为接口协作基线。
