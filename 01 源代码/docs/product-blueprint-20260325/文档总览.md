# 小程序完整业务拆解文档

## 1. 文档目标

这套文档不是对现有 `docs` 的补充，而是基于当前代码现状和本轮业务口径重新整理的一套新版本产品蓝图。目标是解决两个问题：

1. 当前项目到底已经做到哪一步，哪些只是页面原型，哪些还没有形成业务闭环。
2. 后续如何把任务拆成可执行的批次，让不同 AI 或不同开发阶段可以顺序推进。

## 2. 当前结论摘要

基于当前代码判断，这个项目已经具备了“公考刷题小程序基座 + 若干训练页面原型 + 我的页面原型 + 登录壳子 + 基础组件”的基础，但整体仍处于“高保真 Demo / 原型工程”阶段，还没有进入“完整业务闭环”阶段。

已经具备的部分：

- 主体工程结构、分包、基础路由、基础存储工具。
- 首页、练习页、模考页、答题卡、解析页、我的页面的基础界面和部分交互。
- 树形考点组件、答题卡组件、登录入口、基础设置页。
- 行测和申论两个学科的一级入口概念。

还缺失的核心部分：

- 统一的业务模式抽象，当前没有把“每日打卡 / 自主刷题 / 模拟刷题”真正打通。
- 业务接口层仍是通用示例接口，没有落到题库、记录、报告、错题、收藏、每日任务等领域。
- 练习记录、错题本、收藏、刷题报告大多仍停留在 Mock 或本地缓存阶段。
- 申论还没有形成独立业务流，当前只是首页上有一个科目标签。
- 首页和我的页面仍有大量“即将开放”占位逻辑。
- 页面路由和注册存在不一致，说明还没有进入可稳定交付状态。

## 4. 文档用途

你可以把这套文档直接用于：

- 自己梳理完整需求边界。
- 给其他 AI 分发单独任务。
- 作为前后端接口对齐底稿。
- 作为后续迭代验收清单。

## 5. 使用方式建议

推荐做法不是“一次性让 AI 把全部代码写完”，而是每次只给一个任务包。例如：

- 先做“统一答题模式参数设计 + 路由落地”。
- 再做“行测每日一练闭环”。
- 再做“自主刷题闭环”。
- 再做“模拟考试闭环”。
- 最后再做“我的、记录、错题、申论”。

这样每一步都有清晰输入、输出和验收标准，风险最低。
